# Session Memory

## Decisions

- HTTP debug/control panel runs on port `8280`.
- Only one TTS engine stays loaded at a time to conserve VRAM.
- Engine loading is manual from the web UI. Wyoming never auto-loads engines.
- HTML, CSS, and JS are stored outside Python modules.
- Import statements stay at the top of files unless a lazy import is required for optional heavy dependencies.
- Repository hygiene for handoff:
  - `README.md` should stay concise but complete enough to reproduce the stack on a different machine
  - project-only artifacts such as `memory.md` and `implementation_plan.md` stay out of commits via `.gitignore`
  - Docker bind mounts use project-relative `./data/...` paths for easier portability

## Dependency Notes

- A single shared Python environment cannot currently satisfy both `qwen-tts` and `chatterbox-tts` because they pin conflicting `transformers` versions.
- Backend-specific dependencies are therefore split into extras so the core app remains installable and testable.
- `uv` now declares `chatterbox` and `qwen` extras as conflicting, following the official Astral `tool.uv.conflicts` mechanism.
- The Docker/runtime path is now being aligned to CUDA 12.6 rather than CUDA 12.4.
- XTTS still needs compatibility patches at runtime for newer PyTorch/Transformers combinations.
- Qwen3-TTS and Fish Audio integrations are implemented behind lazy imports because their ecosystems are heavy and may evolve independently.

## Risks

- `Fish S2 Pro` may eventually need a dedicated helper runtime for best latency.
- `Qwen3-TTS-Polish-r21` depends on the external `qwen-tts` package and its exact runtime APIs.
- Some engines may work only with GPU and degrade sharply on CPU fallback.
- `Qwen3-TTS` and `Chatterbox` likely need isolated runtimes or helper services if all engines are meant to coexist in one deployed application.

## QA Notes

- QA review on 2026-03-31 found three runtime issues in the manager layer:
  - request serialization used `threading.RLock`, which does not serialize async coroutines correctly on one event loop
  - heavy `load()` / `synthesize()` calls were still running on the event loop
  - `last_voice` / `last_language` leaked across engines because session state stored them globally

  - switching manager serialization to a loop-aware async lock

## Runtime Isolation

- `Chatterbox` and `Qwen` are now wrapped by subprocess-backed isolated engines.
- The main app can point each isolated backend at its own interpreter via `CHATTERBOX_PYTHON` and `QWEN_PYTHON`.
- This is the first step toward truly separate per-engine environments without changing the public HTTP or Wyoming interfaces.
- Isolated workers now have an RPC timeout so a hanging backend does not block the whole service indefinitely.

## Qwen HTTP Runner Fix

- The remaining `Qwen` blocker was narrowed down to integration, not the model runtime itself.
- Direct in-container execution of `QwenTtsEngine` succeeded on CUDA:
  - `load('cuda')` completed successfully
  - `synthesize(...)` completed successfully in about `5.1s`
  - this proved the model/runtime stack was healthy and the hang was outside the engine itself
- The real hang source was the subprocess transport layer:
  - the earlier `IsolatedEngine` / `worker.py` pipe-based RPC remained fragile for `Qwen`
  - active hangs showed the worker blocked in `pipe_write`
  - even after framed responses and file-backed audio payloads, the subprocess IPC approach remained too brittle for this backend
- `Qwen` was therefore moved off the generic pipe IPC path and onto a dedicated local HTTP runner:
  - new `app/engines/http_runner.py` runs an engine in its own localhost HTTP process
  - new `app/engines/http_engine.py` manages that process from the main app
  - `qwen_tts_polish` in `app/engines/registry.py` now uses `HttpEngine` on `127.0.0.1:18001`
- The new runner architecture isolates `Qwen` without sending large synthesis payloads through stdout pipes.
